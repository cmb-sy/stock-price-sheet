# 売買履歴（Trade History）Webアプリ入力・編集機能 設計書

> Date: 2026-06-08
> Status: 承認済み（ユーザー approval 取得済み）

## 目的（Why）

オーナーの**過去の売買記録**と、各取引の**理由**を Web アプリから手入力・編集できるようにする。蓄積された「取引 + 理由」を、将来 AI（Track B スキル）が分析し、売買判断の傾向や反省材料を導けるようにすることが最終目的。

本機能は「取引台帳（trade ledger）」であり、保有銘柄タブのコストベース（取得株価/取得株数）とは独立した記録である。保有銘柄への集計・反映は行わない。

## スコープ境界

### やること
- 新規タブ「売買履歴」への取引行の追加・編集・削除を Web アプリから可能にする。
- 各取引に「理由」を自由記述で残せる。
- 将来の AI 分析結果を書き込むための読み取り専用列「AI分析コメント」を用意する（列のみ。書き込むスキルは本スコープ外）。

### やらないこと（YAGNI / Non-goals）
- AI 分析スキル本体（AI分析コメント列へ書き込む処理）は別プロジェクト。本設計では列を確保するのみ。
- 保有銘柄タブのコストベース（取得株価/取得株数）への自動集計・反映はしない。
- 手数料・約定代金・口座などの追加列は設けない（AI 分析の中心データは「取引の事実 + 理由」であり、過剰な列は目的に貢献しない）。
- CSV インポート等の一括登録はしない。
- AI分析コメント列は Web アプリから編集不可。

## データモデル

### 新規タブ「売買履歴」

- **`config.yaml` には追加しない。** Python 価格エンジンは `config.yaml` のタブのみを対象とするため、売買履歴を含めると価格更新処理が誤って走る。売買履歴は **Web アプリ専用タブ**であり、yfinance Track A / Track B の自動更新対象外。
- 列構成（ヘッダ行 = row 1、左から順）:

| 列名 | 編集可否 | 型 | 必須 | 備考 |
|------|----------|-----|------|------|
| 日付 | manual | text | ✅必須 | 取引日。例 `2026-06-08`。文字列保存（`_coerce` は数値様文字列のみ Number 化、日付は text のまま） |
| 銘柄名 | manual | text | ✅必須 | 行の識別に使用 |
| Ticker | manual | text | 任意 | 例 `7203.T` / `AAPL`。手入力（登録後の自動解決はしない） |
| 売買区分 | manual | select | ✅必須 | `買い` / `売り` の2択 |
| 約定単価 | manual | number | ✅必須 | > 0。数値様文字列は Number 化される |
| 株数 | manual | number | ✅必須 | > 0 |
| 理由 | manual | long text | 任意 | 自由記述。AI 分析の中心データ。textarea 入力 |
| AI分析コメント | read-only | text | — | 将来の Track B スキルが書き込む。Web アプリからは編集不可 |

- `MANUAL_HEADERS['売買履歴'] = ['日付','銘柄名','Ticker','売買区分','約定単価','株数','理由']`
- AI分析コメントは `MANUAL_HEADERS` に含めない → サーバ側 `_resolveManualCol` が書き込みを拒否する。

## バックエンド設計（webapp/Code.gs）

### 1. TABS への追加

```javascript
var TABS = ['保有銘柄', '監視-JP', '監視-US', '売買履歴'];
```

- **末尾に追加する。** `state.holdingsTab = tabs[0]`（= 保有銘柄）が先頭である前提に依存しているため、先頭は変えない。これにより holdings 判定ロジックは無変更で済む。

### 2. `_sheet()` でのタブ自動作成

現状 `_sheet(tabName)` は対象タブが無いと `タブが見つかりません` を throw する。売買履歴タブはオーナーが手動作成しなくても良いよう、**読み取り時に存在しなければ自動作成**する。

- 対象は売買履歴タブのみ（他タブは従来どおり throw）。
- `SpreadsheetApp.openById(...).insertSheet('売買履歴')` で作成し、row 1 に上記の正規ヘッダ行を書き込む。
- **冪等**であること: 既存タブがあれば何もせず返す（ヘッダ行を上書きしない）。複数セッション/リロードで多重作成しない。
- ヘッダ定数は1箇所に定義し、`MANUAL_HEADERS` + AI分析コメント と矛盾しないようにする（正規ヘッダ = `['日付','銘柄名','Ticker','売買区分','約定単価','株数','理由','AI分析コメント']`）。

### 3. `deleteRow` の identity 一般化（重要なバグ修正）

現状の `deleteRow(tabName, rowNum, expected)` は `expected = {ticker, name}` で行の同一性を検証する。**売買履歴では同一銘柄を複数回取引するため、Ticker + 銘柄名 では行が一意に定まらない**（例: 同じ `7203.T` を別の日付・単価で2回購入 → 2行とも ticker/name 一致）。行シフトが起きた際、誤った行を削除し得る。

#### 修正方針
`expected` を「**全 editable 列の表示値マップ**」に一般化する:

- クライアントは削除対象行の編集可能列すべての表示値を `expected = { '<header>': '<displayValue>', ... }` として送る。
- サーバは現在の行を `getDisplayValues()` で読み、`expected` の各キーについて現在値（trim 済み）と突き合わせる。**1つでも不一致なら** `行の内容が変わっています。再読み込みしてください` を throw。
- `expected` が空（キー無し）の場合は拒否する（行番号だけの削除を許さない）。
- ヘッダに存在しないキーが `expected` に含まれる場合は、そのキーの現在値を `''` として突き合わせる。実際の値と一致しなければミスマッチ（= 安全側に倒れて削除拒否）となる。明示的な throw は設けず、ミスマッチ経路に一本化する。
- **戻り値は `true` のみ。セルの値をログ・戻り値に含めない**（PII/ティッカー保護）。
- この一般化は保有・監視タブにも適用される（後方互換: クライアントが全 editable 列を送れば従来より厳密になるだけ）。保有・監視のクライアント側も新しい `ident`（全 editable 列）を送るよう統一する。

## フロントエンド設計（webapp/index.html）— Approach A: 既存 UI 流用 + 限定分岐

売買履歴は保有/監視の二値モデルとは構造が異なる第3のタブ種別。全面的な別 UI は作らず、`isHistoryTab()` による限定的な分岐を既存コードの要所に挿入する。

### 定数追加
- `HISTORY_TAB = '売買履歴'`
- `function isHistoryTab() { return state.tab === HISTORY_TAB; }`
- `SELECT_OPTS = { '売買区分': ['買い', '売り'] }`
- `理由` を NOTES に追加: `{label:'理由', man:true}`（長文編集対象として `LONG_EDIT` にも追加）。

### 分岐ポイント

1. **カードの pill（render, 現状 line 798-799）**
   - 現状: `(isHoldings ? '保有' : '監視')` バッジ。
   - 履歴タブ: 売買区分バッジ（`買い` / `売り`）を表示。`売り`/`買い` で符号色（SIGNED 系の色分け）を流用。
   - 履歴タブでは「売買区分」チップを本文側で重複表示しない（suppress）。

2. **チップ（columnMetrics 相当）**
   - 履歴タブの表示チップ = 日付 / 約定単価 / 株数。

3. **ソート（sortOptions / sortValue）**
   - `sortOptions` に履歴分岐を追加: 並べ替え候補 = 日付 / 約定単価 / 株数 / 銘柄名。
   - `sortValue` に `日付` ケースを追加: `Date.parse()` で数値化（ROLE.updated と同じ手法）。約定単価/株数は数値ソート。
   - デフォルトソートは日付の降順（新しい取引が上）。

4. **入力フィールド（fieldHtml, 現状 line 584-591）**
   - `SELECT_OPTS[label]` が存在する場合は `<select>` を描画（売買区分）。
   - `LONG_EDIT` に含まれる場合は従来どおり textarea（理由）。
   - その他は input。

5. **新規追加（openAdd, 現状 line 929-949）**
   - 現状は Ticker 入力を除外し「Ticker はあとで自動解決します」と表示。
   - 履歴タブ分岐: **Ticker 入力を含める**、登録後自動解決の注記は出さない、`state.required = ['日付','銘柄名','売買区分','約定単価','株数']`。

6. **バリデーション（saveNew / save, 現状 line 962-973）**
   - 現状は 銘柄名 のみ必須チェック。
   - 一般化: `state.required` の全項目が空でないことを検証。
   - 数値チェック: 約定単価 / 株数 は `parseNum` > 0 であること。0・負数・非数値は保存を拒否しエラー表示。
   - これは保有/監視タブにも適用される一般化（required はタブごとに設定済み）。

7. **削除 identity（onDeleteClick, 現状 line 988-1015）**
   - 現状: `ident = { ticker: state.original[ROLE.ticker]||'', name: state.original[ROLE.name]||'' }`。
   - 修正: `ident` = `state.original` の**全 editable 列のコピー**（バックエンドの一般化に対応）。
   - 削除ボタンのラベル: 履歴タブでは「この取引を削除」（保有/監視は「この銘柄を削除」）。確認文言「この操作は取り消せません — 削除する」は共通。

### 触らない箇所
- 用語解説（GLOSSARY_TAB）まわりは無関係。
- 保有銘柄の portfolioHtml（保有専用集計）は履歴タブでは呼ばない。

## エラーハンドリング / 境界値

- 約定単価・株数が 0 / 負 / 非数値 → クライアントで保存拒否しトースト表示。サーバ側 `_coerce` は数値様文字列のみ Number 化するため、非数値は text 保存されるが、クライアント検証で事前に弾く。
- 日付は文字列保存（形式の厳密バリデーションはしない。YAGNI。例 `2026-06-08` を推奨表示）。
- 必須項目欠落 → 保存拒否。
- 削除時の行シフト → 一般化 identity ガードで誤削除を防止。
- タブ未作成 → 初回読み取りで自動作成（冪等）。

## テスト / 検証計画

ローカル検証の限界（clasp/ブラウザ無し環境）を踏まえ、以下を実施:

### 自動検証
- **Code.gs 構文チェック**: `cp webapp/Code.gs /tmp/x.js && node --check /tmp/x.js`。
- **index.html 構文チェック**: 配信ドキュメント再構成（`<?!= include('sortlib'); ?>` を sortlib.html 本体でインライン化 → 全 `<script>` 本文を連結 → `node --check`）。
- **sortlib 単体テスト**: `node --test tests/webapp/test_sortlib.mjs`（既存5件 + 日付ソートを追加するなら拡張）。
- **Python 回帰**: `.venv/bin/python -m unittest discover -s tests`（26件）。売買履歴は config.yaml 非対象なので価格エンジンに影響しないことを確認。

### Adversarial probe（必須）
- **同一銘柄の複数取引の削除**: 同じ銘柄（例 `7203.T`）を異なる日付・単価で2行登録 → 行シフトを発生させた後に片方を削除しようとする → 一般化 identity ガードが内容不一致を検知し、誤った行の削除を拒否することを確認（静的コードレビュー + 可能ならデプロイ後のブラウザ実機）。

### オーナー手動チェック（デプロイ後）
- 売買履歴タブが自動作成され、正規ヘッダで生成される。
- 取引の追加（買い/売り両方）・編集・削除が動作する。
- 理由（長文）が textarea で編集できる。
- AI分析コメント列が読み取り専用（編集 UI に出ない、サーバが書き込み拒否）。
- ソート（日付降順デフォルト、約定単価/株数/銘柄名）が動作する。

### 検証限界の明示
Code.gs サーバ関数・index.html の DOM/inline-JS 配線はデプロイ無しに実行できない。自動検証（構文 + sortlib + Python）+ 安全クリティカルパスの静的 adversarial レビューを実施し、ランタイム確証はオーナーのブラウザチェックリストに委ねる。完全 PASS は主張せず、検証限界を正直に明記する。

## セキュリティ / PII（防御的多重化、緩めない）

- リポジトリは public。コミットされる内容・Actions ログはすべて world-readable 前提。
- 売買履歴の**実データ（実ティッカー・価格・取引・理由）は Google Sheet にのみ存在し、リポジトリには絶対にコミットしない**。
- コミットする Code.gs / index.html には**汎用ヘッダラベル・タブ名のみ**。実ティッカー/価格/PII を含めない。
- 本設計書・コード中のティッカー例は汎用値（`7203.T` / `AAPL`）のみ。
- Code.gs はセル値をログ・戻り値に含めない（`deleteRow` も `true` のみ返す）。
- `webapp/**` への push は本番 clasp デプロイをトリガーする。push 前にユーザー確認を取る。
- allowlist の email は ALLOWED_EMAILS Script Property のみ。コミットファイルに含めない。
